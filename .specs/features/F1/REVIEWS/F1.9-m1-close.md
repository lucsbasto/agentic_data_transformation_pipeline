## Call-graph composition check

`python -m pipeline` → `__main__.cli()` → `cli.ingest.ingest()` → `Settings.load()` → `ManifestDB(settings.state_db_path())` → `compute_batch_identity` → `scan_source` → `validate_source_columns` → `transform_to_bronze` → `collect_bronze` → `write_bronze` → `mark_completed`.

No contract mismatch found. Every import resolves, every return type flows into the next consumer. `LLMClient` is constructed independently (own `LLMCache` + `Settings`) and is not wired into ingest — correct for M1; F2 will call `cached_call` inside the Silver transform.

## Top-3 interface-churn risks for F2

1. **`LLMClient.cached_call` signature** (`src/pipeline/llm/client.py:133-141`). F2 Silver enrichment will likely need batch / streaming modes, structured-output schemas, or tool-use parameters. The current `system`/`user`/`model`/`max_tokens`/`temperature` surface is minimal. Mitigation: `kw_only=True` on `LLMResponse` already protects return-side; adding a `**kwargs` forward or a request-object pattern on the inbound side will be needed.
2. **`transform_to_bronze` function** (`src/pipeline/ingest/transform.py`). A parallel `transform_to_silver` will need the same lineage-column injection (batch_id, source_hash, ingested_at). Extract the pattern into a shared helper before the copy-paste lands.
3. **`ManifestDB` — no `runs` table usage yet** (`src/pipeline/state/manifest.py`). The `runs` DDL is declared in `schemas/manifest.py` but `ManifestDB` has zero methods that write to it. F2/F3 per-layer run tracking will need `insert_run`, `mark_run_completed`, `mark_run_failed`. Additive, not breaking, but the absence means the runs contract is untested.

## Test pyramid assessment

96% coverage is genuine, not inflated. Notable findings:

- **Well covered:** Settings validation (missing key, traversal, bounds), ManifestDB lifecycle (insert, complete, fail, stale recovery, delete, idempotency), LLMClient (miss, hit, retry, fatal, fallback exhaust, invalidate), Bronze transform (schema drift, Enum cast, lineage), CLI e2e (happy, idempotency, missing source).
- **Meaningful hole:** CLI `except PipelineError` branch (`cli/ingest.py:74-80`) is exercised only via Click's own `exists=True` check, not via a `SchemaDriftError` injected mid-pipeline. A test that triggers a schema drift during transform and asserts the CLI exit code + manifest `FAILED` row would close this gap. Add in early F2; not blocking for M1.
- **Acceptable gap:** `LLMCache.open()` `:memory:` branch vs file branch split is shallowly covered. Low risk.

## What must NOT change without a deliberate ADR (carrying into F2)

1. **Bronze `pl.Schema` and Enum closed sets** (`src/pipeline/schemas/bronze.py`). Any column addition, type change, or new Enum member is a schema migration. Pinned by `test_write_bronze_round_trip_preserves_enum_dtype`.
2. **`batches` table DDL** (`src/pipeline/schemas/manifest.py`). Silver/Gold depend on the `batch_id` FK. Column additions OK; renames or type changes require migration SQL.
3. **`compute_cache_key` hash algorithm** (`src/pipeline/llm/cache.py:32-49`). Changing the hash formula invalidates every cached LLM response. Must be versioned if ever altered.
