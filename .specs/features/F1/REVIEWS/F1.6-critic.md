# F1.6 Critic Review — CLI ingest + LLMClient stub

**Verdict: ship with tracked debt.** Both commits are functional and
well-tested for M1 single-operator scope. No critical-now blockers.
Three items will bite hard in later milestones if not addressed.

---

## Critiques

1. **Non-`PipelineError` exceptions leave orphan IN_PROGRESS rows** (later-pain)
   `cli/ingest.py:135` catches only `PipelineError`. A `PolarsError`,
   `OSError`, or `sqlite3.OperationalError` will skip `mark_failed`,
   leave the batch `IN_PROGRESS`, and bubble as a raw traceback —
   the operator sees a Python stack, not a structured message.
   `reset_stale` on next run cleans the orphan, so it self-heals,
   but the error UX is poor and the batch_id is absent from the
   `ClickException` message (`ingest.py:77` — just `str(exc)`).
   **Fix:** add a bare `except Exception` that calls `mark_failed`
   with `error_type="UnexpectedError"`, and prepend `batch_id` to
   the `ClickException` string.

2. **`delete_batch` destroys FAILED-row audit trail** (later-pain)
   `cli/ingest.py:105` deletes the prior FAILED row before re-insert.
   All prior error metadata (type, message, duration, timestamps) is
   lost. There is no `batch_attempts` or history table.
   **Fix (F1.8 or earlier):** add a `batch_history` table, or
   `UPDATE` the existing row to `IN_PROGRESS` (promote in place)
   instead of delete+insert.

3. **`LLMResponse` stub will force a breaking change in F1.7** (later-pain)
   `llm/client.py:28-40` — `LLMResponse` is `frozen=True, slots=True`
   with 5 positional fields. The skill spec (§7) and the docstring
   itself (line 33) promise F1.7 adds `retry_count`, `cost_usd_estimate`,
   and `actual_model`. Adding fields to a `slots=True` dataclass changes
   the constructor signature. Any test using positional construction
   (e.g., `test_llm_client_stub.py:58`) breaks.
   Also: DESIGN.md §6 specifies `__init__(self, settings, cache: LLMCache,
   anthropic_client: Anthropic | None = None)` — shipped stub drops both
   `cache` and `anthropic_client`, accepting `**_: Any` instead.
   **Fix:** add future fields now as `Optional` with defaults (they
   won't break existing callers), and align the constructor with the
   DESIGN signature using a placeholder `cache: Any = None`.

4. **TOCTOU race on idempotency check** (nitpick for M1)
   `cli/ingest.py:93→105→109`: `is_batch_completed` → `delete_batch` →
   `insert_batch` is not atomic. Two concurrent processes on the same
   source will both pass the check; one hits `IntegrityError` wrapped
   as `ManifestError("batch_id already exists")`. Acceptable for M1
   single-operator, but the error message is confusing. Document the
   single-writer assumption.

5. **Duration captured on both paths — confirmed OK** (no action)
   Failure: `ingest.py:136`. Success: `ingest.py:151`. Both use
   `time.monotonic` from the same `started_wall`. Correct.

6. **96-row synthetic fixture vs 153K-row real parquet** (nitpick)
   All integration tests run against the in-memory fixture. The real
   parquet is never exercised in CI before F1.8. Acceptable for M1 —
   schema coverage is high — but add a smoke test against the real
   file before F1.8.

---

## What must survive refactors

- `PipelineError` as the single catch-root in CLI error translation.
- `batch_id` determinism (content hash + mtime).
- `LLMClient` raising `LLMError` (not `NotImplementedError`) so the
  catch-root works uniformly.
- `duration_ms` on both success and failure paths.

## Before F1.7

Verify that `LLMResponse` can be extended without breaking downstream
callers. The `frozen=True, slots=True` dataclass with 5 positional
fields means every new field must have a default or every call site
must be updated. Decide now: keyword-only construction enforced via
`kw_only=True` on the dataclass, or add future fields as `Optional`
with defaults. Either way, the test at `test_llm_client_stub.py:58`
that constructs `LLMResponse` positionally is the canary — if it
doesn't break when you add fields, your strategy is correct.
