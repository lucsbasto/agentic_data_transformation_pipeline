# F1.7 Security Review — LLMCache + LLMClient

**Verdict: LOW risk. No critical or high issues. Two medium items to track.**

| Sev | Summary | Location | Fix |
|-----|---------|----------|-----|
| MEDIUM | `invalidate(prefix=)` LIKE wildcard — `%`/`_` in prefix widen the match | `cache.py:143` | Escape metacharacters: `prefix.replace("%","").replace("_","")` or use `GLOB ?` |
| MEDIUM | Cache-as-oracle — no integrity check on `response_text` read from SQLite | `cache.py:98-105` | Accept risk (local-only DB) or add HMAC column in a future hardening pass |
| LOW | `_first_text_block` returns arbitrary provider string — no length cap or control-char strip before cache/log | `client.py:321-333` | Truncate or sanitize before logging if log-injection matters |
| LOW | `LLMCache` opens with `isolation_level=None` but no WAL pragma — safe but slower under concurrent reads | `cache.py:79` | Add `PRAGMA journal_mode=WAL` like `ManifestDB` does |

## Priority Clearance

1. **Secret handling** — CLEARED. `SecretStr` wraps the key; `get_secret_value()` is called only in `_build_anthropic` (client.py:121) and the raw value is passed to the SDK constructor, never logged. `_redact_secrets` (logging.py:33-46) scrubs any `SecretStr` instance and any key matching `api_key|token|secret|password|auth`. All `self._logger.*` calls in client.py bind only `model`, `cache_key[:16]`, `error_type`, `attempt`, `latency_ms`, token counts — no prompt text, no secret values. The Anthropic SDK `APIError.__str__` emits `message` (server error text), not request headers, so `f"...{exc}"` in LLMCallError is safe.

2. **Prompt-injection / cache collision** — CLEARED. `compute_cache_key` (cache.py:38-56) hashes `model + system + user + max_tokens + temperature` with SHA-256. The key is derived from the full content of all params with unambiguous `field=value\n` framing — no prefix collision is possible across different field boundaries.

3. **Cache-as-oracle** — MEDIUM (accepted risk). If an attacker has write access to `state/manifest.db` they can poison `response_text`. No HMAC or signature protects cached rows. Mitigation: the DB is local-only, not network-exposed, and `pipeline_state_db` rejects `..` traversal (settings.py:56-68). Acceptable for current threat model.

4. **Response content trust** — LOW. `_first_text_block` returns raw provider text. No length cap or control-character filtering. Structlog's `JSONRenderer` will JSON-escape special characters in log output, so ANSI/newline injection into structured logs is neutralized. Cache stores the raw text, which is correct for replay fidelity.

5. **SQL injection in cache** — CLEARED. All queries use `?` parameter binding (cache.py:100, 119-131, 143). The one LIKE query `WHERE cache_key LIKE ?` with `f"{prefix}%"` is parameter-bound but does not escape `%`/`_` metacharacters in `prefix` — a caller passing `%` would match all rows. This is a minor correctness issue (MEDIUM), not exploitable for injection.

6. **Retry/fallback log leakage** — CLEARED. Reviewed all 9 `self._logger.*` call sites (client.py lines 148-302). None bind `system`, `user`, or any `SecretStr`. The `llm.retry`, `llm.fallback`, and `llm.failed` events bind only: `model`, `error_type`, `attempt`, `budget`, `primary`, `fallback`.

## Additional Checks

- **Secrets scan**: No hardcoded keys in `src/`. Test files use obvious dummy values (`sk-abcdefgh`). `.env` is gitignored. `.env.example` has placeholder.
- **Dependencies**: `anthropic>=0.34`, `pydantic>=2.7`, `structlog>=24.1` — no known CVEs at these floors.
- **Command injection**: Zero `exec`/`eval`/`subprocess`/`os.system` in `src/`.
- **Path traversal**: `pipeline_state_db` validator rejects `..` segments.
