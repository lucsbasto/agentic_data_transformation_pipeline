# F1.7 Critic Review — LLMCache + LLMClient

**Verdict: ship as-is for M1.** No critical blockers. Four items to track; two will bite in F2/F3 if ignored.

---

## Findings

### 1. Retry loop is budget + 1, not budget (later-pain)

`client.py:209` loops `range(1, budget+1)` = 3 primary attempts, then `client.py:249` adds one fallback = **4 SDK calls total**. DESIGN.md §6 says "max `PIPELINE_RETRY_BUDGET` (default 3)" without clarifying whether fallback counts. The test at `test_llm_client.py:151-164` asserts `len(calls) == 4` so the *code* is intentional, but the *spec* is ambiguous. Operator expecting "budget=3 means 3 calls max" gets a surprise extra call and one extra cost unit.

**Fix (F2):** Clarify in DESIGN.md: "budget governs primary attempts; fallback is one additional shot outside the budget." Add a log field `total_attempts` to `llm.fallback` event.

### 2. Fallback response cached under primary's key (later-pain)

`client.py:138-144` computes cache key from `primary_model`. When fallback fires (`client.py:249`), the response is stored at `client.py:177` under that same primary-model key. Result: the cached row's `model` field says `qwen3-coder-plus` but the key was derived from `qwen3-max`. A subsequent explicit `cached_call(model="qwen3-coder-plus", ...)` with identical prompts will **miss** — different key, same response. No data corruption, but cost tracking in Silver/Gold will double-count if both paths fire.

**Fix (F2):** Either recompute cache key using `response.model`, or accept the current behavior and document it as "cache key = intended model, not actual model."

### 3. Empty-string response is silently cached (later-pain)

`client.py:321-333` — `_first_text_block` returns `""` when all content blocks are tool-use. This empty string passes `response_text TEXT NOT NULL` in the DDL (`manifest.py:70`) and gets stored forever via `INSERT OR IGNORE`. Silver will receive `""` as a persona label and propagate it without error. F1 only uses text blocks so this is inert today, but F2 tool-use integration will hit it.

**Fix (F2):** Decide policy: raise `LLMCallError("no text block in response")` or return a sentinel. Do not cache `""`.

### 4. FakeAnthropic accepts any kwargs (nitpick)

`test_llm_client.py:51` — `FakeMessages.create(**kwargs)` accepts anything. A new required kwarg in the Anthropic SDK (e.g., `stream` becoming mandatory) would pass CI but fail at runtime. Low risk for M1 since the SDK contract is stable, but worth noting for F4 integration tests.

**Fix (F4):** Add a `_validate_kwargs` check to `FakeMessages` asserting expected keys match `Anthropic.messages.create` signature.

---

## What must survive refactors

- `compute_cache_key` uses `kw_only` args and `temperature:.6f` formatting — changing the format string invalidates every cached row. Treat as a frozen contract.
- `INSERT OR IGNORE` semantics in `LLMCache.put` — the cache never overwrites. Any migration that changes this must flush the table.
- `_RETRYABLE_EXCEPTIONS` / `_FATAL_EXCEPTIONS` tuple ordering — catch blocks rely on tuple membership; additions must go through review.
- `LLMResponse.kw_only=True` — removing it is a breaking change for every caller.
