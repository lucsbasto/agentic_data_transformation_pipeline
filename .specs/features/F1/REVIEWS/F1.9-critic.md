# F1.9 — Adversarial M1-close review (critic)

**Verdict: ship M1. Prototype one thing before F2 (see below).**

---

## Critiques

**1. `runs` table is a hollow organ — F2 will need it but no API exists.** (carrying-forward)
`manifest.py` exposes zero methods for `runs` (no insert, no query, no status update). The DDL reserves `layer IN ('bronze','silver','gold')` but F2 Silver needs `insert_run` / `mark_run_completed` / `get_runs_for_batch` before it can track Silver progress per-batch. If F2 bolts these onto `ManifestDB` ad-hoc, the batch-only transaction pattern (`_transaction` + `_update_status`) will get cloned with subtle divergences. Fix: add the `runs` CRUD in F2's first commit, *before* the Silver transform logic. F2 owns it, not F4.

**2. Two unsynchronized SQLite connections to the same WAL file.** (later-pain)
`LLMCache.open()` and `ManifestDB.open()` each call `sqlite3.connect(target, isolation_level=None)` independently. Neither sets `busy_timeout`. Under F2's 15k-conversation load, the LLMCache writer will occasionally contend with ManifestDB's `BEGIN/COMMIT` blocks. SQLite's default busy timeout is 0 ms — immediate `SQLITE_BUSY`. Single-threaded F1 never hits this; F2's tight cache-write loop will. Fix: set `PRAGMA busy_timeout = 5000` on both connections, or (better) pass a shared connection from the orchestrator.

**3. LLMClient is per-call synchronous — shape is fine, throughput is not.** (later-pain)
`cached_call` blocks for the full network round-trip. With 15k conversations, ~20% cache miss = 3k live calls. At `qwen3-max` median ~800 ms/call, that's ~40 min sequential — blows the <15 min M2 envelope. The `cached_call(system=, user=)` *signature* is correct and should be preserved; the implementation needs a batched/async inner loop. Don't refactor the public API; add an internal `_batch_call` path in F2. Back-of-envelope: 8 concurrent requests brings 40 min → 5 min, within budget.

**4. Cache key is model-scoped, not prompt-version-scoped.** (carrying-forward)
`compute_cache_key` hashes `model + system + user + max_tokens + temperature`. If F2 iterates on the system prompt for persona classification (it will), stale cache rows from the old prompt version silently replay. The operator must manually call `invalidate(prefix=None)` — a full wipe. Fix: add an optional `prompt_version: str` field to the cache key in F2. Cheap, backward-compatible (default to `"v1"`).

**5. `runs` table FK to `batches` means Silver can't track progress if Bronze deletes/retries the batch row.** (nitpick)
`delete_batch` (`manifest.py:233`) does `DELETE FROM batches WHERE batch_id = ?` with no `ON DELETE CASCADE` on `runs`. If a Bronze retry deletes a batch that already has Silver runs referencing it, the FK violation will raise `IntegrityError`. Not a problem today (runs are empty), but wire `ON DELETE CASCADE` or `ON DELETE SET NULL` before F2 writes runs rows.

**6. `ingest.py` catches only `PipelineError` — everything else orphans the batch row.** (nitpick)
Already triaged in INDEX.md F1.6 #5, but worth restating for F2: Silver's `PolarsError` during LLM-response parsing or `httpx.TimeoutError` from the SDK will bypass `mark_failed`. `reset_stale` cleans up on next startup, but the operator gets zero diagnostic context. F2 should widen the catch to `Exception` in the Silver orchestrator.

## Structurally correct — defend against refactor impulses

- **`LLMResponse` as a frozen `kw_only` dataclass.** Future fields (latency, cost, actual_model) add without breaking callers. Do not convert to Pydantic — it's an internal value object, not a serialization boundary.
- **Polars Enum for `direction`, `status`, `message_type`.** The round-trip contract (`test_write_bronze_round_trip_preserves_enum_dtype`) is pinned. Silver must read these as Enum, not re-cast. Do not weaken to Categorical.
- **Deterministic `batch_id` from content hash.** Idempotency is free. Do not add randomness.
- **`LLMCache` separate from `ManifestDB`.** Despite sharing the same `.db` file, the separation of concerns is correct. Merging them into one class would couple batch lifecycle to cache lifecycle.

## The one prototype to de-risk F2

**Prototype a `SilverOrchestrator` that reads one Bronze partition, groups by `conversation_id`, sends 5 conversations to `cached_call`, and writes a single Silver parquet with the LLM-enriched columns (persona, intent).** This forces you to resolve, in ~100 lines, every seam simultaneously: (a) reading Bronze Enum dtypes without re-casting, (b) constructing the per-conversation prompt, (c) discovering the `busy_timeout` contention live, (d) deciding whether `runs` tracking belongs here or is deferred, and (e) measuring actual `qwen3-max` latency to validate the back-of-envelope concurrency estimate. Throw the prototype away after; the learnings shape the F2 DESIGN.md.
