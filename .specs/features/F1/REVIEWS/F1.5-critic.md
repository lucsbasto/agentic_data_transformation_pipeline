# F1.5 Critic Review — Bronze Ingest (adversarial wave 2)

**Ship as-is for M1.** No critical blockers. Four items need attention before Silver/Gold build on top of Bronze output.

---

## Findings

**1. Enum columns lost on parquet round-trip — Silver will re-cast.** (later-pain)
`writer.py:74` re-scans the written parquet but only checks row count. The test at `test_ingest_writer.py:85-92` explicitly documents Polars drops Enum metadata on disk: it compares *values* cast to String, not dtypes. This means every downstream reader (`scan_parquet` in Silver/Gold) sees `String`, not `Enum`. Silver must re-cast or carry a schema-aware reader. Not a bug today, but if Silver assumes Bronze parquet preserves Enums it will silently accept garbage values.
**Action before F1.6:** add a one-line integration test that `scan_parquet` on the Bronze output and asserts `schema["direction"] != pl.Enum(...)` so the contract is explicit and nobody later "fixes" it by accident.

**2. `wrap_cast_error` is dead code in production.** (nitpick)
`transform.py:81-90` exports `wrap_cast_error` via `__all__` (line 97). It is called from exactly one place: the test at `test_ingest_transform.py:80`. No production caller catches `PolarsError` and routes through it — `_ingest` in `test_ingest_e2e.py:56` does a bare `.collect()` with no try/except around the cast. The docstring says "callers ... should catch" but nobody does. Either wire it into the CLI entrypoint in F1.6 or inline it into the test. Leaving an unwired helper exported in `__all__` is a false promise.

**3. Retry-after-FAILED hits PK constraint.** (later-pain)
`test_ingest_e2e.py:35` checks `is_batch_completed`; if False it calls `insert_batch` (line 39). After a FAILED run, `is_batch_completed` returns False, but the `batch_id` row already exists — `insert_batch` raises `ManifestError("batch_id already exists")` at `manifest.py:167-169`. `reset_stale` (line 91 of manifest.py) only sweeps `IN_PROGRESS` to `FAILED`; it does not delete or reset `FAILED` rows. The CLI in F1.6 needs a `delete_or_reset_failed` path, or `_ingest` needs to check `is_failed` and UPDATE instead of INSERT. No test covers re-run after failure.

**4. Atomic-rename window: concurrent reader sees missing directory.** (nitpick, M1-acceptable)
`writer.py:64-66`: `shutil.rmtree(final_dir)` then `tmp_dir.replace(final_dir)`. Between those two calls, `final_dir` does not exist. A concurrent `scan_parquet` hitting that path gets `FileNotFoundError`. Acceptable for M1 single-process, but document the limit so the team does not assume concurrent reads are safe.

**5. Fixture is 96 rows, not 48.** (nitpick)
The commit message says "synthetic 48-row fixture" but `conftest.py` generates 8 * 2 * 3 * 2 = 96 rows. Misleading for anyone reading the commit to size test coverage. Harmless but worth correcting in a subsequent commit message or conftest docstring.

**6. Unknown-enum behavior is a `PolarsError`, not a null-out.** (verified-ok)
Question 1 answered: Polars `cast(pl.Enum(...))` on an unknown value raises `InvalidOperationError` at `.collect()` time. `test_ingest_transform.py:62-84` confirms this. The cast does NOT silently null-out. Good.

---

## What must survive refactors

The layered error vocabulary (`SchemaDriftError` vs `IngestError` vs `ManifestError`) is well-designed and each layer raises the right subclass. `assert_bronze_schema` pre-write guard in the writer, the atomic tmp-dir pattern, `reset_stale` crash recovery, and deterministic `batch_id` from content+mtime are all solid. Keep them.

## Prototype before F1.6

Wire the `wrap_cast_error` path into whatever CLI entrypoint calls `.collect()`. Run it against the real 153K-row parquet with one row poisoned (unknown enum value). Confirm the error message is actionable and the manifest row lands as FAILED with a useful `error_message`. If you skip this, F1.6 will ship a CLI that swallows `PolarsError` as a raw traceback instead of a domain error.
