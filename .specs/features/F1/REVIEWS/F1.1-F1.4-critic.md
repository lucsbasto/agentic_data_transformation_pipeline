# F1.1–F1.4 Adversarial Review

This code is worth building on. The foundation is solid — clean layering, honest scope, real tests. But there are structural decisions that will cost real hours if not addressed before F1.5.

## Critiques

**1. ManifestDB has no crash recovery — IN_PROGRESS rows are orphans forever.**
If the process is killed between `insert_batch` (state/manifest.py:115) and `mark_completed`/`mark_failed`, the batch row stays `IN_PROGRESS` permanently. There is no `recover_stale_batches()`, no heartbeat, no timeout. The F4 agent loop will retry the same batch_id, hit the `IntegrityError` on line 130, and halt. This is not "later pain" — the ingest flow in F1.5 will hit it the first time `Ctrl-C` lands at the wrong moment.
- *Severity*: **critical-now**. Add a `reset_stale(older_than_seconds: int)` method or allow `insert_batch` to overwrite `FAILED`/stale `IN_PROGRESS` rows.

**2. `runs.status` has no CHECK constraint — unlike `batches.status`.**
`schemas/manifest.py:38` declares `status TEXT NOT NULL` with no `CHECK`. The `batches` table (line 18) has `CHECK (status IN ('IN_PROGRESS','COMPLETED','FAILED'))`. Any typo in a future `runs` INSERT silently succeeds. The `runs` table also has zero indexes, so querying by `batch_id` or `status` will table-scan.
- *Severity*: **later-pain**. Add `CHECK (status IN (...))` and `CREATE INDEX idx_runs_batch ON runs(batch_id);` now while the table is empty.

**3. Enum closure with zero growth room — `conversation_outcome` is the silent bomb.**
`schemas/bronze.py:22-35` freezes `direction`, `status`, `message_type` as `pl.Enum`. The design doc (DESIGN.md:107) says "Any new value triggers a schema-drift error." But `conversation_outcome` is kept as `pl.String` (line 54) because it has 7 Portuguese labels. The real risk is `message_type`: WhatsApp regularly adds types (`reaction`, `poll`, `order`). When that happens, Bronze ingest hard-fails with no fallback. There is no "unknown" bucket, no alerting, no soft-fail mode.
- *Severity*: **later-pain**. Document the explicit upgrade path (add value to tuple, re-deploy). Consider adding an `_unknown` sentinel to each Enum and casting unrecognized values to it, logging a warning. Decide before F1.5.

**4. `test_errors.py` and `test_schemas_manifest.py` test constants, not behavior — they pass even if production code is gutted.**
`test_errors.py` (lines 17-28) asserts `issubclass(ConfigError, PipelineError)`. This tests the class hierarchy declaration, not any runtime behavior. `test_schemas_manifest.py:101-108` asserts `BATCH_STATUS_IN_PROGRESS == "IN_PROGRESS"` — a tautology. These inflate coverage without catching regressions. The `test_enums_reject_unknown_value` in `test_schemas_bronze.py:74-84` uses a bare `except Exception` with `# pragma: no cover`, which means the test passes both when Polars raises AND when it doesn't (if the exception path is uncovered).
- *Severity*: **nitpick** (for the constant tests), **later-pain** (for the bare except pattern — pin the exception type to `polars.exceptions.InvalidOperationError` or at minimum `pl.ComputeError`).

**5. DESIGN.md describes 12 implementation tasks; only 5 shipped (steps 1–5). Six modules are specced but missing.**
DESIGN.md section 12 lists `ingest/reader.py`, `ingest/transform.py`, `ingest/writer.py`, `cli/ingest.py`, `llm/client.py`, `llm/cache.py`. None of these exist yet. The `__main__.py` prints "none registered yet." This is fine for an incremental plan, but the commit messages say "feat(F1)" as if F1 is progressing — anyone reading the git log will think F1 is further along than it is.
- *Severity*: **nitpick**. Consider labeling these commits F1.1–F1.4 explicitly in the message body.

**6. "Self-healing" has zero seeds in M1.**
The project description says "self-healing pipeline." The DESIGN.md mentions the agent loop in F4 and reserves the `runs` table. But there is no retry logic in ManifestDB, no dead-letter tracking, no health-check endpoint, no metric emission. The word "self-healing" is marketing until F4 ships something that actually recovers from a failure without human intervention. The closest thing is the idempotent batch check (DESIGN.md step 4), which is "don't repeat work" — not "heal from failure."
- *Severity*: **nitpick** for now. Becomes **critical** if the README repeats this claim.

## What the implementer got right

The separation between `schemas/manifest.py` (pure DDL strings) and `state/manifest.py` (runtime class) is clean and will survive refactoring. The `_transaction()` context manager with explicit `BEGIN`/`ROLLBACK`/`COMMIT` over `isolation_level=None` is correct and avoids the sqlite3 module's notoriously confusing implicit transaction behavior. The Bronze schema is measured from actual data, not assumed — the docstring in DESIGN.md section 1 with real cardinalities is the kind of evidence that prevents schema churn. Tests for the ManifestDB (insert, complete, fail, duplicate, rollback, file-backed) exercise the real sqlite3 engine with `:memory:`, not mocks — this is the right boundary to test at.

## Before F1.5 (ingest): read, measure, prototype

Read the Polars `pl.Enum` casting behavior for unknown values — confirm it raises `ComputeError` (not `SchemaError` or `InvalidOperationError`) so your `SchemaDriftError` catch in `ingest/transform.py` targets the right exception. Measure the actual parquet write time for 153K rows with `zstd` compression to confirm the single-partition strategy won't bottleneck. Prototype the atomic-rename pattern (DESIGN.md section 4, step 126: "writes to a temp dir first, then atomic-renames") on your target filesystem — `os.rename()` is not atomic across mount points, and if `data/bronze/` is on a different volume than `/tmp`, the rename will fail silently or copy-then-delete. Use `pathlib.Path.rename()` and test it. Finally, decide whether `batch_id = sha256(hash + mtime)[:12]` gives you enough collision resistance at 12 hex chars (48 bits) — for 10K batches the birthday bound is comfortable, but document the assumption.
